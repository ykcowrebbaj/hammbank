1、判断题：RDD的transformation算子中的函数使用到了外部的对象需要序列化，是因为对象需要从driver端发送到executor端

对。是因为对象需要从driver端发送到远程的executor端。

2、谈谈spark编程，遇到数据倾斜，如何定位原因？

1）根据log日志信息定位。尤其可以关注distinct、groupByKey、reduceByKey、aggregateByKey、join、cogroup、repartition等算子，产生的相应的日志。

2）分析定位法。数据倾斜发生在第几个stage，可以在任务运行的过程中，通过Spark Web UI界面，观察到每一个stage中运行的task的数据量以及task的运行时间，从而进一步确定是不是task分配的数据不均匀导致了数据倾斜。

3）若task发生内存溢出（OOM），直接看yarn-client模式下本地log的异常栈，或者是通过YARN查看yarn-cluster模式下的log中的异常栈。一般来说，通过异常栈信息就可以定位到你的代码中哪一行发生了内存溢出。然后在那行代码附近找，一般也会有shuffle类算子，此时很可能就是这个算子导致了数据倾斜。

4）查看导致数据倾斜的key的数据分布情况。我们可以先对key采样样本数据，然后统计出每个key出现的次数来获取key的数据的分布情况。这主要是为之后选择哪一种技术方案来解决数据倾斜提供依据。

3、简述spark中的统一（动态）内存管理和静态内存管理的区别？

1）静态内存管理模块只包含堆内内存(On-heap Memory)。统一内存管理模块包括了堆内内存(On-heap Memory)和堆外内存(Off-heap Memory)两大区域。

2）静态内存管理方式早就已经被淘汰，spark新的版本都是直接使用统一内存管理（动态内存管理）。

3）在 Spark 最初采用的静态内存管理机制下，存储内存、执行内存和其他内存的大小在 Spark 应用程序运行期间均为固定的，用户需要在应用程序启动前进行配置。而在目前的统一内存管理中， Execution 内存和 Storage 内存可以互相共享的。也就是说，如果 Execution 内存不足，而 Storage 内存有空闲，那么 Execution 可以从 Storage 中申请空间；反之亦然。

4、判断题：spark中，最开始的未经优化的HashShuffleManager，假设map task有M个，reduce task个数有R个，那么，每个reduce的buffer需要从M*R个磁盘小文件拉取数据

错。最开始的未经优化的HashShuffleManager会产生大量的小文件，每个 Map Task 为每个 Reduce Task 生成一个文件，通常会产生大量的文件（即对应为 M*R 个中间文件，其中 M 表示 Map Task 个数，R 表示 Reduce Task 个数） ,也就是说每个reduce的buffer需要从M个磁盘小文件拉取数据。
